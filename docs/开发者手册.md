# CRA AI Assistant 开发者手册

> 版本：4.0.3 | 最后更新：2026-04-12

---

## 1. 项目概述

CRA AI Assistant 是一款基于 Electron 的桌面应用程序，用于辅助临床研究助理（CRA）从临床试验文档（方案文件、受试者文件）中自动提取入排标准、访视计划、受试者数据，并通过 AI 进行受试者资格核验。

### 1.1 技术栈

| 技术 | 版本 | 用途 |
|------|------|------|
| Electron | ^33.3.1 | 桌面应用框架 |
| React | ^18.3.1 | UI 框架 |
| TypeScript | ^5.7.2 | 类型系统 |
| Webpack | ^5.97.1 | 模块打包 |
| Zustand | ^4.5.5 | 状态管理 |
| Tailwind CSS | ^3.4.17 | 样式框架 |
| pdf-parse | ^1.1.1 | PDF 文本解析 |
| pdf-to-img | ^6.0.0 | PDF 转图片（ESM） |
| axios | ^1.7.9 | HTTP 客户端 |
| electron-store | ^8.2.0 | 持久化配置存储 |

### 1.2 系统要求

- Node.js >= 18
- npm >= 9
- Windows 10 / macOS 10.15+ / Linux

## 2. 快速开始

### 2.1 安装依赖

```bash
npm install
```

### 2.2 环境配置

复制 `.env.example` 为 `.env` 并填入 API 密钥：

```bash
cp .env.example .env
```

编辑 `.env`：

```env
ZHIPU_API_KEY=your_zhipu_api_key_here
ZHIPU_API_ENDPOINT=https://open.bigmodel.cn/api/paas/v4/chat/completions
```

### 2.3 开发运行

```bash
# 方式一：一键启动（推荐）
npm run start
# 此命令会：启动渲染进程开发服务器 → 构建主进程 → 启动 Electron

# 方式二：分步运行
npm run build:main           # 构建主进程
npm run build:renderer       # 启动渲染进程开发服务器 (端口 3000)
npm run electron             # 启动 Electron（需先构建主进程）
```

### 2.4 生产构建

```bash
npm run build    # 构建主进程 + 渲染进程
```

### 2.5 其他命令

| 命令 | 说明 |
|------|------|
| `npm run dev` | Watch 模式（主进程 + 渲染进程，不启动 Electron） |
| `npm run clean` | 清理 `dist/` 目录 |

## 4. 开发指南

### 4.1 双进程架构

本项目遵循 Electron 安全最佳实践：

- **主进程**（`src/main/`）：运行在 Node.js 环境，拥有文件系统和操作系统 API 访问权限。负责 AI 调用、文件处理、配置管理。
- **渲染进程**（`src/renderer/`）：运行在 Chromium 环境，`nodeIntegration: false`，`contextIsolation: true`。只能通过 `contextBridge` 暴露的 `electronAPI` 与主进程通信。
- **预加载脚本**（`src/preload/`）：安全桥接层，使用 `contextBridge.exposeInMainWorld` 暴露有限的 IPC 方法。
- **共享代码**（`src/shared/`）：类型和常量，两个进程均可导入。

### 4.2 Webpack 配置

项目有两套独立的 Webpack 配置：

**主进程**（`webpack.main.config.js`）：
- 导出一个**数组**，同时构建主进程入口和预加载脚本
- 目标：`electron-main` 和 `electron-preload`
- 路径别名：`@main → src/main`，`@shared → src/shared`
- 输出：`dist/main/index.js` 和 `dist/preload/index.js`

**渲染进程**（`webpack.renderer.config.js`）：
- 目标：`web`（不是 `electron-renderer`）
- 开发服务器端口：3000
- 路径别名：`@shared → src/shared`，`@renderer → src/renderer`
- CSS Pipeline：`style-loader → css-loader → postcss-loader`（Tailwind）

### 4.3 TypeScript 配置

两套独立的 `tsconfig`：

| 配置文件 | 目标环境 | 模块系统 | 说明 |
|----------|---------|----------|------|
| `tsconfig.json` | Node.js | CommonJS | 主进程 + 预加载脚本 |
| `tsconfig.renderer.json` | 浏览器 | ESNext | 渲染进程，包含 DOM 类型 |

### 4.4 状态管理

使用 Zustand 单一 Store + `persist` 中间件：

- 存储键：`cra-ai-assistant-storage`（localStorage）
- 持久化内容：设置、所有工作表数据数组
- 不持久化：`isProcessing`、`processingStage`、`processingProgress`、`isTesting`、`testResult`、`error`
- **恢复逻辑**：启动时自动将序列化的日期字符串还原为 Date 对象，并清除过期的资格核验结果

```typescript
// 从组件中使用
import { useStore, useSettings, useActiveWorksheet } from './hooks/useStore';

const settings = useSettings();
const activeWorksheet = useActiveWorksheet();
const criteria = useStore(state => state.inclusionCriteria);
```

### 4.5 添加新的 IPC 通道

1. 在 `src/shared/constants/ipc-channels.ts` 中定义通道名
2. 在 `src/main/ipc-handlers.ts` 中注册 `ipcMain.handle()`
3. 在 `src/preload/index.ts` 中通过 `contextBridge` 暴露方法
4. 在 `src/renderer/types.d.ts` 中更新 `Window.electronAPI` 类型声明
5. 在渲染进程中通过 `window.electronAPI.xxx()` 调用

### 4.6 添加新的工作表

1. 在 `src/shared/types/worksheet.ts` 中定义数据类型和 `WorksheetType`
2. 在 `src/shared/constants/app.ts` 中添加 `WORKSHEET_CONFIG` 条目
3. 在 `src/renderer/hooks/useStore.ts` 中添加状态和 CRUD 操作
4. 在 `src/renderer/components/worksheet/` 中创建组件
5. 在 `src/renderer/App.tsx` 中注册路由
6. 在 `src/renderer/components/layout/Sidebar.tsx` 中添加导航项

### 4.7 添加新的 AI 适配器

1. 在 `src/main/agents/gateway/adapters/` 中创建新适配器，继承 `BaseAdapter`
2. 实现 `chat()` 和 `chatWithImage()` 方法
3. 在 `src/main/agents/gateway/router.ts` 的 `SmartRouter` 中注册
4. 在 `src/shared/types/config.ts` 中添加对应的配置字段
5. 在 `src/renderer/components/settings/SettingsDialog.tsx` 中添加 UI

## 5. AI 集成

### 5.1 模型配置

| 模型 | 类型 | 用途 |
|------|------|------|
| glm-5.1 | 文本模型 | 文本数据提取、标准分析 |
| glm-4.6v-flash | 视觉模型（免费） | 扫描版 PDF、图片 OCR |
| gpt-4o-mini | 文本模型（OpenAI） | OpenAI 兼容端点 |
| gpt-4o | 视觉模型（OpenAI） | OpenAI 兼容图片处理 |

### 5.2 路由逻辑

`SmartRouter` 根据以下规则选择模型和适配器：

```
contentType === 'text'  → textModel  (glm-5.1 / gpt-4o-mini)
contentType === 'image' → visionModel (glm-4.6v-flash / gpt-4o)
provider === 'zhipu'    → ZhipuAdapter
provider === 'openai'   → OpenAIAdapter
```

### 5.3 图片 API 注意事项

调用 GLM 视觉模型时，base64 编码的图片必须包含 data URI 前缀：

```typescript
// ✅ 正确
image_url: { url: `data:image/png;base64,${base64Image}` }

// ❌ 错误（缺少前缀会导致 API 错误）
image_url: { url: base64Image }
```

### 5.4 PDF 处理策略

| 文件类型 | 检测方式 | 处理策略 |
|----------|---------|---------|
| 文本型 PDF | pdf-parse 提取文本 > 50 字符 | 分块 → 文本提示词 → AI |
| 扫描型 PDF | pdf-parse 提取文本 ≤ 50 字符 | pdf-to-img 转图片 → 图片提示词 → AI |
| 图片文件 | 文件扩展名判断 | 直接 base64 编码 → 图片提示词 → AI |

**分块策略**：文本按约 2000 token 分段，500 token 重叠，在段落/句子边界处切分。

## 6. 配置管理

### 6.1 配置优先级

```
环境变量 (.env) > 加密存储 (electron-store) > 默认值 (DEFAULT_CONFIG)
```

### 6.2 API 密钥安全

- API 密钥通过 `electron.safeStorage.encryptString()` 加密
- 加密后存储在 `electron-store` 中（额外的 `encryptionKey` 保护）
- `.env` 文件已加入 `.gitignore`，不会被提交到版本控制
- 主进程的 `getConfig()` 从不返回加密密钥的明文给渲染进程

### 6.3 默认配置

```typescript
{
  apiEndpoint: 'https://open.bigmodel.cn/api/paas/v4/chat/completions',
  apiKey: '',
  textModel: 'glm-5.1',
  visionModel: 'glm-4.6v-flash',
  provider: 'zhipu',
}
```

## 8. 安全注意事项

| 安全措施 | 实现方式 |
|----------|---------|
| 进程隔离 | `contextIsolation: true`，`nodeIntegration: false` |
| API 密钥保护 | `electron.safeStorage` 加密 + `electron-store` 加密存储 |
| 文件路径验证 | 防止路径遍历攻击（禁止 `..`），校验 MIME 类型 |
| 文件大小限制 | 最大 100MB |
| IPC 通信安全 | 仅暴露必要的 API 方法，主进程验证所有输入 |
