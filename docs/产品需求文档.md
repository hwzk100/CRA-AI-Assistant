# CRA AI Assistant 产品需求文档 (PRD)

> 版本：4.1.0 | 最后更新：2026-04-29

---

## 1. 产品概述

### 1.1 产品定位

CRA AI Assistant 是一款面向临床研究助理（Clinical Research Associate, CRA）的桌面辅助工具，利用 AI 大模型技术，自动从临床试验方案文件和受试者资料中提取关键数据，并进行受试者资格核验，大幅减少 CRA 的手工操作。

### 1.2 目标用户

| 角色 | 使用场景 |
|------|---------|
| 临床研究助理（CRA） | 从方案文件中提取入排标准，核验受试者是否符合试验要求 |
| 临床研究协调员（CRC） | 录入受试者信息，查看访视计划和用药记录 |
| 项目经理（PM） | 快速查看试验项目的入排标准和核验结果 |

### 1.3 核心价值

- **自动化提取**：AI 自动从 PDF/图片中提取入排标准、访视计划、受试者数据，替代手工录入
- **智能核验**：AI 逐条比对入排标准与受试者数据，给出资格判定和证据引用
- **多格式支持**：支持文本型 PDF、扫描型 PDF、图片（PNG/JPG）等多种文件格式
- **数据管理**：以工作表形式集中管理所有提取数据，支持查看和编辑

## 2. 功能需求

### 2.1 系统设置

#### 2.1.1 AI 服务配置

**需求描述**：用户可配置 AI 服务的连接信息，支持多个 AI 服务商。

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 智谱 AI 配置 | API Key、API 端点、文本模型、视觉模型 | P0 |
| OpenAI 兼容配置 | API Key、API 端点、模型名称 | P1 |
| 连接测试 | 一键测试当前配置的 API 是否可用，显示延迟和模型信息 | P0 |
| 配置持久化 | 设置自动保存，重启应用后保留 | P0 |
| 重置默认 | 支持一键恢复默认配置 | P1 |

**默认配置**：
- 服务商：智谱 AI
- 端点：`https://open.bigmodel.cn/api/paas/v4/chat/completions`
- 文本模型：`glm-5.1`
- 视觉模型：`glm-4.6v-flash`

#### 2.1.2 API 状态指示

**需求描述**：在应用顶部实时显示 API 连接状态。

| 状态 | 显示 | 条件 |
|------|------|------|
| 已连接 | 绿色圆点 + "API 已连接" | API Key 已配置（非空） |
| 未配置 | 红色圆点 + "API 未配置" | API Key 为空 |

### 2.2 文件上传与管理

#### 2.2.1 文件上传

**需求描述**：用户可通过拖拽或点击上传临床试验相关文件。

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 方案文件上传 | 上传临床试验方案文件（PDF/图片） | P0 |
| 受试者文件上传 | 上传受试者资料文件（PDF/图片） | P0 |
| 拖拽上传 | 支持将文件拖拽到指定区域 | P0 |
| 点击上传 | 点击区域弹出文件选择对话框 | P0 |
| 多文件上传 | 同一区域可上传多个文件 | P1 |
| 文件格式校验 | 仅接受 PDF、PNG、JPG、JPEG 格式 | P0 |
| 文件大小校验 | 单文件最大 50MB | P0 |
| 文件状态显示 | 显示每个文件的状态：待处理/处理中/已完成/失败 | P0 |

#### 2.2.2 文件处理

**需求描述**：上传后自动进行文件类型检测和预处理。

| 检测结果 | 条件 | 后续处理 |
|----------|------|---------|
| 文本型 PDF | pdf-parse 提取文本 > 50 字符 | 分块文本 → AI 文本提取 |
| 扫描型 PDF | pdf-parse 提取文本 ≤ 50 字符 | PDF 转图片 → AI 文档分类 → 差异化提取 |
| 图片 | 文件扩展名为 png/jpg/jpeg | AI 文档分类 → 差异化提取 |

#### 2.2.3 受试者文档智能分类

**需求描述**：受试者文件上传后自动分类每页文档类型，根据分类使用差异化提取策略。

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 自动分类 | AI 视觉模型轻量分类（仅返回类别标签，约 20 token） | P0 |
| 差异化提取 | 根据类别使用针对性提取 prompt，提升提取精度 | P0 |
| 分类标签展示 | 文件列表显示彩色分类标签（病历=蓝、化验单=绿、药物表=琥珀、其他=灰） | P1 |

**分类类别**：

| 类别 | 识别特征 | 提取侧重 |
|------|---------|---------|
| 病历 | 主诉、诊断、病史、医嘱、体格检查、入院/出院记录 | 人口学 + 病史 + 生命体征 + 医嘱 |
| 化验单 | 检验项目、数值、单位、参考范围、异常标志 | 检验数据（存入 vitalSigns） |
| 药物表 | 药品名称、规格、批号、有效期、库存数量 | 药品信息（存入 medications） |
| 其他 | 不属于以上三类的文档 | 通用提取（兜底） |

**分类流程**：

```
扫描型 PDF: 转图片 → 逐页分类(轻量AI) → 按类别分组 → 差异化提取 → 合并
图片: 直接分类 → 选对应prompt → 提取
文本型 PDF: 不分类，直接使用通用 prompt
```

### 2.3 AI 数据提取

#### 2.3.1 入排标准提取

**需求描述**：从方案文件中自动提取入选标准和排除标准。

**输入**：方案文件（PDF 或图片）

**输出**：
- 入选标准列表：每条包含序号、内容、来源分类
- 排除标准列表：每条包含序号、内容、来源分类

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 自动分类 | AI 自动区分入选标准和排除标准 | P0 |
| 编号保留 | 保留原文档中的编号 | P1 |
| 多文件合并 | 多个方案文件的结果自动合并去重 | P1 |
| 多数投票 | 同一标准被多次提取时，取多数结果 | P2 |

#### 2.3.2 访视计划提取

**需求描述**：从方案文件中自动提取访视计划信息。

**输入**：方案文件（PDF 或图片）

**输出**：
- 访视计划列表：每条包含访视名称、时间窗口、检查项目列表

#### 2.3.3 受试者数据提取

**需求描述**：从受试者文件中自动提取关键数据。

**输入**：受试者文件（PDF 或图片）

**输出**：
- 人口学信息：受试者编号、姓名缩写、年龄、性别、民族
- 生命体征：血压、心率、体温、身高、体重等
- 病史：既往疾病、手术史、过敏史
- 用药记录：药品名称、剂量、频率、起止日期、适应症

---

### 2.4 资格核验

#### 2.4.1 自动资格核验

**需求描述**：AI 逐条比对入排标准与受试者数据，给出资格判定。

**触发条件**：受试者数据提取完成后，若已有入排标准数据，自动触发核验。

**输出**：
- 逐条核验结果：每条标准的 pass/fail/unknown 状态
- 证据引用：引用受试者数据中支持判定的具体信息
- 置信度：0-1 的置信度分数
- 总体结论：受试者是否符合入排标准
- 核验摘要：文字总结

| 核验规则 | 说明 |
|----------|------|
| 入选标准 | 必须**全部通过**才判定为符合 |
| 排除标准 | 必须**全部不满足**才判定为符合 |
| 无法判断 | 数据不足以判定时标记为 unknown |

#### 2.4.2 核验结果展示

**需求描述**：在工作表中直观展示核验结果。

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 入选标准核验标签 | 绿色"符合" / 红色"不符合" | P0 |
| 排除标准核验标签 | 绿色"不满足"（未被排除）/ 红色"满足"（应被排除） | P0 |
| 核验原因 | 鼠标悬停或展开显示 AI 引用的证据 | P1 |
| 清除核验结果 | 一键清除所有核验结果，保留标准数据 | P1 |

## 4. 数据规格

### 4.1 支持的文件格式

| 格式 | 扩展名 | MIME 类型 | 说明 |
|------|--------|-----------|------|
| PDF | `.pdf` | application/pdf | 文本型/扫描型自动检测 |
| PNG | `.png` | image/png | 图片文件 |
| JPEG | `.jpg`, `.jpeg` | image/jpeg | 图片文件 |

### 4.2 文件限制

| 限制项 | 值 |
|--------|-----|
| 单文件最大体积 | 50MB（UI）/ 100MB（主进程验证） |
| 单区域最大文件数 | 100 |
| 接受的文件格式 | PDF, PNG, JPG, JPEG |

### 4.3 AI 模型规格

| 参数 | 值 |
|------|-----|
| 文本请求超时 | 120 秒 |
| 图片请求超时 | 180 秒 |
| 默认温度 | 0.1 |
| 默认最大 Token | 4096 |
| 文本分块大小 | 动态计算，基于模型上下文窗口的 60%，上限 6000 tokens（约 9000 中文字符） |
| 分块重叠 | 分块大小的 10%（约 600 tokens） |
| 请求间隔 | 连续 API 调用至少间隔 1 秒 |
| 429 重试 | 最多 3 次，指数退避（2s → 4s → 8s） |

### 4.4 模型规格表

| 模型 | 上下文窗口 | 单次最大图片数 | 说明 |
|------|-----------|--------------|------|
| glm-5.1 | 128,000 | - | 默认文本模型 |
| glm-4.6v-flash | 8,192 | 1 | 默认视觉模型（免费） |
| glm-4v-plus | 128,000 | 5 | 高级视觉模型 |
| gpt-4o | 128,000 | 10 | OpenAI 视觉模型 |
| gpt-4o-mini | 128,000 | 10 | OpenAI 文本模型 |

## 5. 非功能需求

### 5.1 性能

- 应用启动时间 < 5 秒
- 文件上传响应 < 1 秒（不含 AI 处理）
- UI 操作响应 < 200ms
- 支持 50MB 以下的 PDF 文件处理

### 5.2 安全

- API 密钥使用 `electron.safeStorage` 加密存储
- 渲染进程无 Node.js 访问权限（`nodeIntegration: false`）
- 上下文隔离（`contextIsolation: true`）
- 文件路径防遍历攻击
- 所有来自渲染进程的数据在主进程中验证

### 5.3 兼容性

- Windows 10+
- macOS 10.15+
- 主流 Linux 发行版

### 5.4 可靠性

- 配置数据自动持久化到 localStorage
- 应用重启后恢复工作表数据
- AI 调用失败时自动重试（429 限流最多 3 次指数退避）
- 连续 API 调用间隔至少 1 秒，避免触发限流
- PDF 解析结果缓存在内存中，避免重复解析
- 文本型 PDF 提取前按关键词预筛选，减少无效 AI 调用
- 文件处理失败不影响其他已处理数据
